# M10 Multi-session ASR

状态：设计草案；下一步实现计划的输入。M10 目标是在一次 recording 内按本地 VAD 把 ASR 拆成多个 provider session，减少按音频时长计费 provider 的静音成本，同时不牺牲最终 ASR 文本。

## 1. 目标

- 一次 recording 仍然对应一条 history record 和一个可选 WAV。
- `asr.sessions[]` 从当前固定 1 条变成 1..N 条。
- 本地 VAD 判定用户暂停说话时，让当前 ASR session 收尾；用户重新说话时开新 session。
- provider 是否启用该能力由 provider 私有配置决定，例如 `~/.config/shuohua/asr/doubao.toml` 里的 `idle_pause = true`。
- VAD 参数属于全局 voice 配置，例如 `~/.config/shuohua/config.toml` 的 `[voice.vad]`。
- release history 保持最终结果导向，不记录 partial、VAD frame、provider debug event；开发评估走 trace sidecar。

## 2. 非目标

- 不改 history schema 字段结构，不升 version。
- 不把 VAD trace 写进 history。
- 不让 Apple 默认启用 idle pause；Apple 可以实现同名配置，但默认关。
- 不做跨 session 文本去重的复杂算法。第一版依赖小 overlap、provider finalization 和现有 post chain。

## 3. VAD 选择

M10 默认候选是 Silero VAD。WebRTC VAD 和 RMS 已在真实环境里对呼吸、键盘、风扇类场景不稳，不作为默认。Apple SpeechDetector 暂不作为 Doubao 省费路径的一部分。

当前实测结论：

- Silero 能识别明显停顿，连续说话样本不会乱切。
- `threshold = 0.5`、`pause_silence_ms = 1500`、`pre_roll_ms = 300` 是当前默认候选。
- 长停顿、弱语音、短句恢复仍有风险样本，必须先 shadow trace，再正式切 session。

## 4. 配置边界

### Provider 配置

每个 provider 自己加载私有配置，新增同名字段：

```toml
idle_pause = false
finalize_timeout_ms = 5000
```

字段语义：

- `idle_pause`：允许 voice 层用本地 VAD 控制该 provider 的 session 收尾和重启。
- `finalize_timeout_ms`：voice 发出 `is_last=true` 后，最多等 provider 返回 final segment / `Done` 多久。

建议默认值：

- Doubao：默认 `false`，用户显式打开后启用。它按音频时长计费，适合该能力。
- Apple：默认 `false`。本地不计费，通常没有必要切 session，但保留同一配置语义。

该配置不是 provider capability 矩阵，也不是全局 voice 配置。provider 收到 voice 层的普通 `send_pcm([], true)` / `close()` / 下一次 `open()`，按现有 trait 语义完成收尾和重启。

`finalize_timeout_ms` 是 provider 私有字段，不是 VAD 参数；这个等待时间取决于 provider 的网络、协议和本地 helper 行为。

### VAD 配置

VAD 参数放在全局 `config.toml`：

```toml
[voice.vad]
backend = "silero"
threshold = 0.5
pause_silence_ms = 1500
pre_roll_ms = 300
max_overlap_ms = 200
min_start_voiced_frames = 2
```

这些参数描述的是本地 VAD 如何切 recording timeline，应该能迁移到其他按音频时长计费的 streaming ASR。provider 私有配置只决定该 provider 是否启用这套外部 idle pause 控制，以及它自己的 finalize 等待策略。

## 5. 控制协议

### Active

当前有 ASR session。recorder 持续产出 16kHz mono s16le PCM；voice 层同时：

- 把 PCM 喂给当前 ASR session；
- 把 PCM 喂给 Silero；
- 把 PCM 写入 replay ring buffer；
- 累计当前 session 的真实发送样本数。

### Pause

Silero 连续静音达到 `pause_silence_ms`：

1. voice 发送 `send_pcm([], true)` 给当前 ASR session。
2. 等 provider 发完 final segment / `Done`。
3. 记录该 session 的 `text`、`started_at`、`ended_at`、`audio_ms`。
4. close 当前 session。
5. 进入 Idle；麦克风和 VAD 继续运行，不再喂 ASR。

Silero 认为停止了，就让 ASR 收尾。ASR 不负责否决 pause；ASR 只负责把当前 session finalize。

### Resume

Idle 中 Silero 检测到 speech：

1. 从 ring buffer 取 pre-roll。
2. 允许小 overlap，避免 VAD 边界附近的词被丢掉。
3. 新开 provider session。
4. 先发送 replay PCM，再发送当前和后续 PCM。
5. 回到 Active。

推荐初始参数：

```text
pre_roll_ms = 300
max_overlap_ms = 200
min_start_voiced_frames = 2
pause_silence_ms = 1500
```

## 6. Timeline 与 overlap

M10 要用 sample timeline 管理 PCM，而不是靠 wall-clock 猜。

建议 recorder/voice 内部使用：

```rust
struct PcmChunk {
    start_sample: u64,
    samples: Vec<i16>,
}
```

voice 层维护：

```text
last_sent_sample
ring_buffer
current_session_start_sample
current_session_sent_samples
```

resume 时：

```text
desired_start = speech_start_sample - pre_roll_samples
send_start = max(desired_start, last_sent_sample - max_overlap_samples)
```

允许小 overlap 的理由：VAD 边界附近的 100-300ms 可能属于上一句尾巴，也可能属于下一句开头。让两个 ASR session 都看到少量上下文，比严格不重叠更不容易丢首字/尾字。

限制：

- overlap 只发生在 resume pre-roll。
- `max_overlap_ms` 第一版不超过 200。
- 不做大段重复发送。
- 第一版不做跨 session 文本去重；如果 trace 证明短词重复明显，再单独设计。

## 7. History 语义

字段结构不变，但 M10 后需要放宽当前不变量。

- `sessions[].started_at` / `ended_at`：该 ASR session 在 recording timeline 上实际发送音频的起止时间。
- `sessions[].audio_ms = ended_at - started_at`。
- `asr.audio_ms = Σ sessions[].audio_ms`。
- 因为 resume overlap，`sessions[]` 允许相邻时间区间重叠。
- 因为重复发送 overlap，`asr.audio_ms` 允许略大于 `duration_ms`。

这不需要升 schema version，因为字段没变；只是当前 `sessions.len() == 1` 和不重叠假设在 M10 落地时解除。

## 8. Trace 验证

开发期用 `voice.vad_trace = true` 和 `--features dev-vad-trace` 生成 sidecar：

```text
~/.local/state/shuohua/traces/<recording_id>.jsonl
```

trace 用于验证：

- VAD active 区间是否覆盖 ASR segment/utterance 区间。
- pause/resume 是否过碎。
- overlap 是否导致明显重复文本。
- `recording_end.vad.saved_ms` 是否达到预期。
- provider finalization 延迟是否可接受。

release history 不消费 trace；trace 文件可随时删除。

## 9. 错误策略

- 新 session `open()` 失败：本次 recording 进入 error，提示用户，不静默丢音频。
- 当前 session `send_pcm([], true)` 失败：进入 error，保留已收到 segment。
- 等 `Done` 超时：按 provider 的 `finalize_timeout_ms` 关闭 session 并标记本次 recording error。第一版不继续开新 session，避免在 provider 状态未知时静默丢音或重复提交。
- provider 未开启 `idle_pause`：完全走当前单 session 路径。

## 10. 实现顺序

1. Provider 配置：Doubao / Apple 都支持 `idle_pause` 字段，默认 false。
2. Voice VAD 配置：新增 `[voice.vad]`，包含 backend、threshold、pause/resume/overlap 参数。
3. PCM timeline：把 voice 内部改成 sample-indexed chunk 和 ring buffer。
4. Silero backend：从 dev trace/probe 中提取正式 backend，保留纯 controller 单测。
5. Session controller：实现 Active / Idle / Pausing / Opening 状态。
6. History writer：从 `SegmentCapture` 折叠一条 session 改成累积多条 session。
7. Trace/report：继续验证 coverage、saved ratio、session count 和 overlap。

## 11. 当前开放问题

- `max_overlap_ms = 200` 是否足够，需更多短句恢复样本验证。
- `pause_silence_ms = 1500` 是否会在真实长停顿场景漏弱语音，需继续 trace。
